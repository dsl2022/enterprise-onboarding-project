What it does

Self-service where it used to be a ticket queue

Large orgs onboard apps to SSO and grant access through slow, ticket-driven, error-prone manual work. This platform makes both self-service and auditable — request, review, approve, and provision for real, with a record of every step.

App onboarding

Register an internal app. On approval it provisions a real Entra app registration and returns the client ID — no manual portal clicking.

Access requests

Browse a catalog and request access. On approval it adds a real Entra group membership. My Access shows what you hold and lets you request removal.

Teams & review queue

Delegate ownership with teams, and triage everything in one worklist — onboarding and access requests share a single review queue.

Tamper-evident audit

Every action is hash-chained and immutable — a verifiable record of who requested, who decided, and what the system executed.

The server is always authoritative. The UI only hides what you can't do — it never decides what you're allowed to do.

Request lifecycle

One request engine, one state machine

Onboarding and access share the same engine. Provisioning is asynchronous — a worker drives transitions through a transactional outbox with retry and backoff. Separation of duties is enforced: the requester can never be the approver.

// onboarding

DRAFT→ SUBMITTED→ UNDER_REVIEW approve→ APPROVED→ PROVISIONING create app reg→ ACTIVE
from UNDER_REVIEW: reject→REJECTED request changes→CHANGES_REQUESTED resubmit→SUBMITTED

// access

SUBMITTED→ UNDER_REVIEW approve→ APPROVED→ PROVISIONING add group member→ GRANTED
also: reject→REJECTED request changes→CHANGES_REQUESTED · removal runs the same flow in reverse
Security & authorization

Authorization is deterministic, layered, and enforced server-side

Roles set the floor; ownership, separation of duties, and an immutable audit chain sit on top of every action.

RBAC from Entra app-roles

Roles come from Entra app-roles, not group claims. Permissions are the union of all roles you hold, at the most-permissive scope.

ABAC ownership

"Own"-scoped actions only ever touch resources you own — a role grants the verb, ownership scopes the object.

Separation of duties

*.decide is rejected with 403 if the principal is the request's requester or submitter — regardless of role.

Impersonation, guarded

A Super Admin's capabilities reduce to the impersonated role — but identity, SoD, ABAC, and audit always resolve to the real principal. You can't launder a self-approval. Audit records actor = real, effectiveRole = impersonated.

Immutable, hash-chained audit

Each row is H(prevHash ‖ row), written by a single-writer outbox relay. The app's DB role has INSERT + SELECT only — UPDATE/DELETE are revoked and trigger-guarded. It literally cannot rewrite history.

Verify the chain

GET /audit/verify recomputes the whole chain and returns { valid, checkedThrough, brokenAt } — integrity is checkable on demand, not assumed.

    Safety by design

    • The model is not a security boundary. All authorization stays deterministic, outside the model.
    • Model output is untrusted input — validated against the real catalog and policy, never trusted.
    • Least privilege, bound to the user — the assistant can never do more than the person it's helping.
    • Human-in-the-loop for any future write-action, and fail closed when unavailable.
    • Auditable — "AI suggested X" ≠ "human Y decided X" ≠ "system executed X."
    • Designed to deny the lethal trifecta — private data, untrusted content, and an exfiltration path are never present together.
